# Plateforme Web — v1.5.0

> **Système de mise à jour robuste + installeur en un fichier.** La plateforme sait désormais se mettre à jour toute seule depuis le panneau admin sans jamais risquer de se casser (application atomique par échange de dossier, contrôle de démarrage de la nouvelle version, rollback automatique si elle ne démarre pas), gère la compatibilité plugin ↔ plateforme (contrainte liste/range déclarée dans chaque `plugin.json`), et peut être **installée en un seul fichier `install.php`** qui télécharge et configure toute la plateforme depuis GitHub en quelques clics. Ce cycle bump la ligne mineure (Y) car il intègre un nouveau sous-système majeur.

## [ADDED]
- **Auto-update robuste du core (Blue-Green Staging Swap)** ([dev_server.py](../dev_server.py)) — l'ancienne application « fichier par fichier en place » est remplacée par : téléchargement de l'artefact de release curé (`lumen3d-web-<version>.zip`, fallback zipball source) → vérification `SHA256SUMS` + intégrité de l'archive → matérialisation de l'arbre complet dans un dossier de staging → **contrôle de démarrage hors-ligne** de la nouvelle version (`dev_server.py --check` en sous-processus) → **basculement atomique** par renommages `os.replace` journalisés (via un processus superviseur détaché, exécuté depuis `%TEMP%` pour ne verrouiller aucun fichier) → **sonde de santé en ligne** (`/api/health`) → **rollback automatique** si la nouvelle version ne démarre pas. Un journal de pivot sur disque rend chaque état réconciliable : une interruption (kill, coupure de courant) est réparée au prochain démarrage (`_reconcile_pivot`) — soit finalisée, soit intégralement restaurée. Suppressions upstream gérées gratuitement (l'arbre stagé est bâti uniquement depuis l'archive) ; données utilisateur, identifiants, statistiques et plugins mis à jour indépendamment sont préservés (liste `_UPDATE_PROTECT` étendue + garde de provenance).
- **Endpoint `/api/health`** ([dev_server.py](../dev_server.py)) — sonde publique liveness + version (`{ok, web, server}`), consommée par le garde-fou du superviseur de pivot et réutilisable par tout moniteur externe. Rapporte aussi le résultat de la dernière mise à jour (`done`/`rolled_back`) à travers le redémarrage.
- **Compatibilité plugin ↔ plateforme (liste/range)** ([js/core/compat.js](../js/core/compat.js), [dev_server.py](../dev_server.py), [api/_admin_lib.php](../api/_admin_lib.php)) — nouveau champ optionnel `platformCompat` dans `plugin.json`, acceptant une **plage** (`">=1.4.0 <2.0.0"`, `"^1.4.0"`, `"~1.4.1"`) ou une **liste** (`["1.3","1.4.x"]`, `["1.4.0","1.4.1"]`, `["1.x"]`). Trois résolveurs jumeaux (JS, Python, PHP) validés contre **un même vecteur de conformité de 42 cas** ([tests/compat-vector.json](../tests/compat-vector.json)). Sémantique fail-safe : champ absent ⇒ compatible ; illisible ⇒ **incompatible** (fail-closed) ; version plateforme inconnue ⇒ gate désactivé. Les 16 plugins livrés déclarent `"1.x"`.
- **Gating de compatibilité bidirectionnel + rapport de pré-vol** ([dev_server.py](../dev_server.py) `_update_preflight_report`, [api/admin.php](../api/admin.php)) — à la découverte, un plugin incompatible est exclu (son `index.js` n'est jamais chargé) ; avant une mise à jour core, l'endpoint `update_preflight&target=X.Y.Z` classe les plugins installés en `ok` / `willQuarantine` / `blocking` (ce dernier réservé au cas « plus aucun mode de rendu compatible »). Ordre d'application : **core d'abord**.
- **Isolation totale du sous-système plugin** ([js/core/plugin-registry.js](../js/core/plugin-registry.js), [js/pages/viewer.js](../js/pages/viewer.js)) — registre de quarantaine (`getQuarantined()`) alimenté à chaque rejet (meta invalide, id ≠ dossier, placement incohérent, incompatible, `index.js` en échec, `implement()` manquant, `init()` throw) ; `try/catch` par plugin dans `buildToolbarButtons`/`activate`/`deactivate`/`initAll` ; barrière `try/catch` globale autour de toute la phase plugin dans `viewer.js`. Un plugin cassé, incompatible ou malveillant **ne peut plus jamais empêcher le viewer 3D de démarrer** (règle 1.1).
- **UX admin — onglet Mises à jour** ([js/pages/admin/tab-updates.js](../js/pages/admin/tab-updates.js), [css/admin-shell.css](../css/admin-shell.css)) — stepper de phases live (`preflight → backup → download → verify → staging → verifying → planning → pivoting → restart`), rapport de pré-vol avec impact plugins avant confirmation, suivi à travers le redémarrage via la sonde `/api/health`, bannière de résultat (succès / rollback automatique) acquittable.
- **UX admin — badges de compatibilité plugins** ([js/pages/admin/tab-plugins.js](../js/pages/admin/tab-plugins.js)) — un plugin incompatible affiche un badge ambré « incompatible » + la raison, bascule figée, et une note explicative (réactivation automatique dès qu'une mise à jour satisfait sa contrainte).
- **Installeur autonome en un fichier** ([install.php](../install.php)) — page d'installation élégante (assistant multi-étapes, barres de progression, reprise sur erreur) qui vérifie les prérequis, télécharge la dernière release depuis GitHub (dépôt **codé en dur**, aucune URL contrôlable par l'utilisateur), extrait avec validation anti-zip-slip par entrée, configure le mot de passe admin (format PBKDF2 interopérable avec les serveurs Python/PHP), puis **s'auto-verrouille** (refuse de s'exécuter si un credential existe) et propose son auto-suppression. Téléchargement par tranches avec reprise (Range) pour survivre aux limites d'exécution des hébergements mutualisés.
- **Pipeline de release CI** ([.github/workflows/release.yml](../.github/workflows/release.yml), [.github/workflows/ci.yml](../.github/workflows/ci.yml), [tools/check_version.py](../tools/check_version.py), [tools/build_release.py](../tools/build_release.py)) — sur tag `v*` : garde de cohérence version (`max(changelog) == tag`, monotonie, unicité) → contrôle de démarrage (`dev_server.py --check`) → build de l'artefact curé (**allowlist** runtime) + `version.json` (empreinte sha256 par fichier) + `SHA256SUMS` → création de la release avec les notes du changelog. CI sur chaque push/PR : cohérence version + `--check`.
- **Contrôle de démarrage hors-ligne `--check`** ([dev_server.py](../dev_server.py)) — valide un arbre plateforme (compilation de `dev_server.py`, présence des entrypoints, JSON des langues, cohérence `version.json`/changelog, plugins) et sort 0/1 avec un rapport JSON. Sert de garde-fou de pivot, de test CI et d'outil manuel.

## [OPTIMIZED]
- **Détection de mise à jour via l'artefact curé** ([dev_server.py](../dev_server.py) `_update_check`) — préfère l'asset de release `lumen3d-web-<version>.zip` (avec `version.json` + `SHA256SUMS`) au zipball source brut ; télécharge par tranches avec vérification `Content-Length` (une archive tronquée échoue au téléchargement, plus jamais silencieusement à l'application).
- **Rangement du dépôt** ([.gitattributes](../.gitattributes), [.gitignore](../.gitignore), [changelog/](../changelog/)) — `.gitattributes export-ignore` retire les fichiers non-runtime (README, `.github/`, `preprocess/`, `DOCS/`, `tools/`, …) des archives source GitHub ; 181 anciens changelogs (0.x → 1.2.x) déplacés dans `changelog/archive/` (ignoré du calcul de version, inclus dans l'artefact) ; `.gitignore` complété (`version.json`, `dist/`, état d'installeur, `.claude/settings.local.json`).

## [FIXED]
- **Course TOCTOU sur le lancement de mise à jour** ([dev_server.py](../dev_server.py) `_start_update`) — le check-then-set de `_UPDATE_STATE["running"]` est désormais sous `_UPDATE_LOCK` : deux POST `update_apply` concurrents (double-clic, deux onglets) ne peuvent plus lancer deux pipelines.
- **Échecs de copie avalés en silence** ([dev_server.py](../dev_server.py)) — l'ancien `except OSError: pass` par fichier (backup ET application) est supprimé : une sauvegarde incomplète (fichier verrouillé, disque plein) **abandonne** la mise à jour avant tout écrasement au lieu de rapporter un faux succès ; le backup est vérifié (`testzip`) après écriture.
- **Redémarrage aveugle après update** ([dev_server.py](../dev_server.py)) — l'ancien `os.execv` inconditionnel (qui laissait le site éteint si la nouvelle version ne démarrait pas) est remplacé par le basculement garde-fou avec sonde de santé et rollback automatique.
- **Fichiers périmés jamais supprimés** ([dev_server.py](../dev_server.py)) — l'application par échange d'arbre supprime désormais les fichiers retirés en amont (calculés par diff des manifestes `version.json`), au lieu de les laisser traîner et se faire redécouvrir.

### Correctifs issus de la revue de sécurité pré-publication (7 défauts confirmés + corrigés)
- **Rollback interrompu finalisé à tort** ([dev_server.py](../dev_server.py) `_reconcile_pivot`) — une restauration interrompue laissait `phase="applied"` ; la réconciliation au démarrage pouvait alors « finaliser » un arbre à moitié restauré. Corrigé : phase `rolling_back` journalisée **avant** toute restauration + la réconciliation ne roule en avant que si l'arbre vivant correspond **intégralement** au manifeste `version.json` (sha256 par fichier), la version du changelog seule n'étant plus une preuve. Régression couverte ([tests/test_update_reconcile.py](../tests/test_update_reconcile.py)).
- **Sonde de santé sur hôte `0.0.0.0`** ([dev_server.py](../dev_server.py) `_loopback_for`) — un serveur lancé avec `--host 0.0.0.0` provoquait un rollback systématique (on ne peut pas *se connecter* à `0.0.0.0` — WinError 10049). La sonde et l'attente de port mappent désormais l'adresse d'écoute joker vers `127.0.0.1`.
- **Course enfant/superviseur sur le journal de pivot** ([dev_server.py](../dev_server.py) `_spawn_server`/`main`) — le serveur relancé par le superviseur exécutait `_reconcile_pivot` et consommait le journal en pleine sonde. Corrigé par la variable d'environnement `LUMEN_SKIP_PIVOT_RECONCILE`.
- **Résolveur compat PHP fail-open sur objet JSON** ([api/_admin_lib.php](../api/_admin_lib.php)) — un `platformCompat` en forme d'objet (`{"min":"1.4"}`) était accepté par PHP (tableau associatif) alors que les jumeaux JS/Python le rejettent. Corrigé (rejet des tableaux non-séquentiels) + cas de régression ajouté au vecteur.
- **`plugins.php` public cassé sur PHP 7.4/8.0** ([api/_admin_lib.php](../api/_admin_lib.php), [api/datasets.php](../api/datasets.php)) — le `require_once` du lib partageant un type de retour `: never` (8.1+) faisait un 500 de la découverte de plugins sur le plancher PHP annoncé. Type retiré (comportement inchangé).
- **Plugin canal qui throw cassait le boot du viewer** ([js/components/channel-panel.js](../js/components/channel-panel.js), [js/pages/viewer.js](../js/pages/viewer.js)) — les points d'extension `getChannelUI`/`bindChannelUI`/`syncUI` étaient invoqués hors `try/catch` : un plugin canal défaillant faisait échouer tout le boot. Isolés par plugin + filet de sécurité `catch` au niveau du boot, restaurant la garantie « un plugin cassé ne casse jamais le viewer ».

## Notes
- **Sécurité (à conserver en tête)** : le code d'un plugin (`index.js`) s'exécute avec les pleins privilèges de la page. Le champ `platformCompat` gate la *fonctionnalité*, jamais la *confiance* — tant qu'aucune couche de bac à sable/signature n'est livrée, la seule source de plugins autorisée reste le `js/modules/` curé par l'opérateur (pas de feed tiers arbitraire). `install.php` code le dépôt GitHub en dur (aucune URL utilisateur) et valide chaque entrée d'archive contre le zip-slip.
- **Tests** : cœur du pivot (application avant/arrière + rejeu idempotent après crash), résolveur de compatibilité sur 42 cas dans les **trois** langages (Python/JS/PHP contre le même vecteur), `--check`, logique de pré-vol (quarantaine + blocage), et validation navigateur du chargement de `compat.js`, des endpoints `/api/health` + `/api/plugins` (filtrage compat) et du montage du panneau admin. Voir [DOCS/update-system/](../DOCS/update-system/).
- **Versionnage** : `dev_server.py:__version__` 0.13.0 → 0.14.0 (version de l'outil serveur, distincte de la plateforme). Cette v1.5.0 est la première release GitHub publiée.
