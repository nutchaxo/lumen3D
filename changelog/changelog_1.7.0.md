# Plateforme Web — v1.7.0

> **Durcissement — clôture des limitations v1 de sécurité.** Ce cycle reprend une à une les limitations explicitement listées dans les *Notes* de [v1.6.0](changelog_1.6.0.md) et les résout : **authenticité de release par signature Ed25519** (nouveau sous-système cryptographique, vérifieur pur-Python vendored + libsodium côté installeur), **CSP stricte sur les hôtes PHP/statiques**, **verrouillage du contexte-élément de `style-src`** (fin de `'unsafe-inline'` pour les `<style>` injectés), **durcissement du nonce** (source `currentScript.nonce`, retrait du `<meta>` lisible), et **complétion de la lane bac à sable** (révocation à chaud, événements hôte→frame, pont d'état workspace, toggle piloté par événement). Bump de la ligne mineure (Y) : nouveau sous-système d'authenticité majeur.

## [ADDED]
- **Authenticité de release — signature Ed25519 (L7)** ([ed25519_pure.py](../ed25519_pure.py), [dev_server.py](../dev_server.py), [install.php](../install.php), [tools/build_release.py](../tools/build_release.py), [tools/gen_signing_key.py](../tools/gen_signing_key.py), [.github/workflows/release.yml](../.github/workflows/release.yml)) — un **vérificateur Ed25519 pur-Python** (RFC 8032, stdlib seule, aucune dépendance) est livré avec la plateforme. La CI signe le `SHA256SUMS` de la release avec une graine privée (secret GitHub `LUMEN_SIGNING_KEY`) → asset `SHA256SUMS.sig`. Avant d'appliquer une mise à jour, le serveur re-vérifie cette signature détachée contre une **clé publique épinglée** dans le code courant (jamais issue du téléchargement) : chaîne `clé épinglée → sig → SHA256SUMS → sha256 → zip`. L'installeur `install.php` fait de même via **libsodium** (intégré à PHP ≥ 7.2). Clé vide = « non configuré » (intégrité sha256 seule, averti bruyamment) ; **clé posée = signature OBLIGATOIRE, fail-closed** (une release non signée ou mal signée est refusée). Amorçage : `python tools/gen_signing_key.py` génère la paire, à coller dans les deux constantes épinglées + le secret CI.
- **Événements hôte→frame du bac à sable (L3)** ([js/core/plugin-sandbox.js](../js/core/plugin-sandbox.js), [js/pages/viewer.js](../js/pages/viewer.js)) — la capability `events.subscribe`, déclarable mais inerte en v1.6, est désormais **émise** : `viewer.js` pousse `camera`/`render`/`channels-updated` vers `PluginSandbox.emit`, fan-out debouncé et **projeté en whitelist** (camera → `{position,target}` nombres seuls, jamais d'objet THREE) aux seules frames abonnées.
- **Pont d'état workspace du bac à sable (L4)** ([js/core/plugin-sandbox.js](../js/core/plugin-sandbox.js)) — un plugin sandboxé peut persister/restaurer son état via `saveState()` (SDK frame) → RPC `state` → cache hôte, relu par `getState()` du shim et re-poussé par `set-state` au respawn, sans jamais exposer le DOM parent.

## [OPTIMIZED]
- **CSP stricte sur hôtes PHP & statiques (L1)** ([api/_html_server.php](../api/_html_server.php), [_serve.php](../_serve.php), [.htaccess](../.htaccess), [router.php](../router.php), [fast_server.py](../fast_server.py)) — l'injection de nonce par requête + l'en-tête CSP enforcing, jusque-là réservés au serveur dev Python, sont portés aux hôtes PHP (`lumen_serve_html` : realpath path-contained, substitution `{{CSP_NONCE}}`, en-têtes CSP + `X-Frame-Options` + `no-store`) via un rewrite Apache (`.htaccess`→`_serve.php`) et `router.php` (php -S), et au `fast_server.py` (import de `_csp_policy`, source unique). Ferme la brèche « les hôtes PHP n'avaient pas de CSP » (le gate de confiance client y était contournable).
- **`style-src` — contexte-élément verrouillé (L8)** ([dev_server.py](../dev_server.py), [api/_html_server.php](../api/_html_server.php), toutes les `*.html`, [js/core/colorblind.js](../js/core/colorblind.js)) — `style-src-elem` passe à `'self' 'nonce-…' fonts.googleapis.com` **sans** `'unsafe-inline'` : un `<style>` injecté (vecteur CSS fort — redressing pleine-page, `@import`) est bloqué ; les `<style>` propres portent le nonce par requête. `style-src-attr` conserve `'unsafe-inline'` pour les ~200 `style=""` data-driven (largeurs, couleurs de canaux) dont la seule forme CSP-propre serait de la CSSOM/utility-class disproportionnée — le résidu est CSS-seul (script-src est déjà nonce-locked) et l'exfiltration par `url()` est déjà fermée par `img-src`/`connect-src 'self'`. `style-src` reste en repli CSP2.
- **Nonce durci — source `currentScript.nonce` (L9)** ([js/core/plugin-registry.js](../js/core/plugin-registry.js), [js/core/plugin-sandbox.js](../js/core/plugin-sandbox.js), [js/core/colorblind.js](../js/core/colorblind.js), toutes les `*.html`) — le `<meta name="csp-nonce" content="…">` **mondialement lisible** est supprimé de toutes les pages. Les scripts qui consomment le nonce le lisent depuis **leur propre balise** (`document.currentScript.nonce`, capturé au chargement) ; leurs `<script>` portent `nonce="{{CSP_NONCE}}"`. Le *nonce-hiding* du navigateur (attribut DOM vidé après parse, propriété IDL `.nonce` conservée) empêche l'exfiltration par sélecteur d'attribut CSS, contrairement à un `<meta content>`.
- **Révocation à chaud des plugins sandboxés (L2)** ([js/core/plugin-registry.js](../js/core/plugin-registry.js), [js/pages/viewer.js](../js/pages/viewer.js)) — le `trustEpoch` est désormais **consommé** : un watcher sonde `/api/health`, et à tout changement d'epoch re-fetche `/api/plugins` et **détruit à chaud** (`PluginSandbox.kill` + retrait du bouton + quarantaine) toute iframe d'un plugin dont l'approbation a été révoquée — sans attendre le rechargement du viewer.
- **Toggle du bac à sable piloté par événement (L5)** ([js/core/plugin-sandbox.js](../js/core/plugin-sandbox.js)) — l'état visuel d'un bouton toggle sandboxé (`_applyButtonState`) reflète le `lastToggle` de la frame au lieu d'un retard d'un clic.
- **Portée des placements du bac à sable — décision d'architecture (L6)** ([js/core/plugin-registry.js](../js/core/plugin-registry.js), [DOCS/plugin-sandbox/SPEC.md](../DOCS/plugin-sandbox/SPEC.md)) — motif de quarantaine spécifique par placement : `shaders` **jamais** sandboxable (GLSL compilé synchroniquement, exécuté GPU par frame — aucune frontière RPC async possible ; in-page trust obligatoire, permanent) ; `channels` différé par conception (l'API channel-panel remet le nœud DOM vivant au plugin — un schéma déclaratif + capabilities d'effet-canal serait nécessaire, sans consommateur actuel : le design est esquissé pour le jour où un channel plugin tiers apparaît).

## [FIXED]
- **Signature binaire brute corrompue par `strip()`** ([ed25519_pure.py](../ed25519_pure.py) consommateurs, [dev_server.py](../dev_server.py), [install.php](../install.php)) — une signature Ed25519 de 64 octets peut légitimement commencer/finir par un octet égal à un blanc ASCII ; le parsing tolérant du format ne `strip()` plus jamais une charge brute de 64 octets (le hex canonique reste le format CI). Détecté par [tests/test_release_signature.py](../tests/test_release_signature.py).

## Notes
- **Vérification systématique** : chaque correctif a été vérifié — vecteurs RFC 8032 exacts + chaîne Python↔PHP (signature produite par `ed25519_pure` acceptée par `sodium_crypto_sign_verify_detached`, tampering rejeté des deux côtés) ; gate de signature de l'updater fail-closed (clé absente/sig manquante/sig invalide/mauvaise clé/vérifieur absent) ; CSP servie et **enforcée en navigateur réel** sur serveur Python ET hôte PHP (un `<style>` non-noncé est bloqué, les `style=""` fonctionnent, la modale colorblind injectée s'applique via `currentScript.nonce`, `<meta csp-nonce>` absent, aucun placeholder fuité) ; broker RPC (émission d'événement projetée, cache d'état) ; suite complète **25/25 verte** (19 py + 3 js + 3 php).
- **Limitations acceptées de l'updater (statuées, hors-scope v1)** : (1) **auto-update PHP** — le pipeline de bascule Blue-Green vit dans `dev_server.py` ; un hébergement pur-PHP applique les mises à jour via `install.php` (re-téléchargement complet), pas de swap à chaud supervisé ; (2) **pas de delta** — une mise à jour télécharge l'artefact complet (allowlist curée), pas un diff par fichier (simplicité + intégrité globale par sha256 ; l'artefact reste petit) ; (3) **mode git** — sur un checkout `.git`, l'updater refuse de toucher au contenu dev (`_UPDATE_PROTECT`), il vise les déploiements de release.
- **Authenticité — état d'amorçage & clé épinglée** : `_RELEASE_PUBKEY_HEX` (dev_server) et `$PINNED_PUBKEY` (install.php) sont **vides par défaut** → intégrité sha256 seule, averti. La signature devient obligatoire dès que la clé publique est posée. **Important** : poser la clé **dans la source du dépôt puis commiter** (elle ship ainsi dans chaque release) — `dev_server.py` n'est **pas** dans `_UPDATE_PROTECT`, donc une clé posée uniquement sur un hôte déployé serait **écrasée par la mise à jour suivante**. C'est cohérent avec le modèle « l'opérateur est aussi l'éditeur » (le lab publie ses propres releases) : la clé, versionnée, se propage à chaque release et survit aux updates. Propriété d'amorçage (TOFU) : la **première** release qui introduit la clé n'est authentifiée que par sha256 (le serveur courant, encore sans clé, ne peut pas vérifier une signature qu'il n'épingle pas encore) ; toutes les suivantes le sont par signature, fail-closed.
