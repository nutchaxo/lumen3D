# Plateforme Web — v1.8.0

> **White-label — fondations (Objectif 1, phase socle).** Premier jalon de la généralisation « white-label » : la plateforme cesse d'être câblée pour un domaine unique et gagne une **couche de configuration d'instance** (le « contenu de l'étude ») que le moteur générique lit au boot, séparée du code. Cette version pose la **fondation transverse** — store `config/` public, singleton client `InstanceConfig`, injection serveur `{{SITE:…}}` (Python **et** PHP), endpoint `/api/site.php` (jumeaux Python/PHP), substitution de tokens i18n, protection contre l'auto-update. Le contenu métier (textes des pages, terminologie, blocs d'accueil, marque) est migré vers cette couche dans les versions suivantes. Bump de la ligne mineure (Y) : nouveau sous-système majeur. Voir [DOCS/whitelabel/PLAN.md](../DOCS/whitelabel/PLAN.md).

## [ADDED]
- **Couche de configuration d'instance — store `config/` public** ([config/instance.json](../config/instance.json), [config/defaults/neutral/instance.json](../config/defaults/neutral/instance.json)) — un nouveau dossier `config/` **servi comme `lang/*.json`** (public, jamais sous `api/` qui est bloqué) porte l'identité de l'instance : marque (`brand.name/monogram/logoEmoji/tagline/organization`), vocabulaire de l'échantillon (`specimen.singular/plural`), SEO (`seo.description/keywords`), titres de page, pied de page, navigation. Un **défaut neutre** (`config/defaults/neutral/`) sans aucune présomption biologique est livré pour les nouvelles installations ; le `config/instance.json` committé porte l'identité IRIBHM actuelle (aucune régression pour le déploiement en place). Les secrets ne vivent JAMAIS ici — ils restent sous `api/`.
- **Singleton client `InstanceConfig`** ([js/core/instance-config.js](../js/core/instance-config.js)) — IIFE classique (nom nu, pas d'ESM ni `window.`) chargé **avant** i18n. Expose `load()`, `get(path)`, `tokens()`, `applyDom(root)`, `applyHead()`. Deux canaux de consommation : (1) **substitution de tokens i18n** — `I18n.t()` interpole `{brand}`, `{specimen}`, `{specimenPlural}`, `{org}`, `{tagline}`… depuis la config, pour que les fichiers de langue restent neutres ; (2) **binding DOM** — `[data-instance="path"]` (textContent) et `[data-instance-attr="attr:path"]` (attributs), jumeau client de l'injection serveur. Défaut neutre embarqué (fail-soft : la page rend même hors-ligne / avant le wizard).
- **Endpoint `/api/site.php` — jumeaux Python + PHP** ([dev_server.py](../dev_server.py), [api/site.php](../api/site.php)) — `GET ?action=get&doc=<instance|theme|legal|pages/<slug>>` (lecture publique) ; `POST ?action=save|reset|publish&doc=…` (session admin + CSRF obligatoires). Écritures **atomiques** (temp + `os.replace` / tempnam + rename) en **0644** (public, lisible par n'importe quel serveur statique — pas 0600 comme les secrets `api/`). Résolution de nom de doc durcie (slug `^[a-z0-9][a-z0-9_-]{0,63}$`, familles fixes) : traversal rejeté. `reset` restaure le défaut neutre ; `publish` promeut `draft`→`published` (constructeur de pages, à venir). **PHP est la cible de déploiement prioritaire** : `api/site.php` est une implémentation de premier plan (helpers `_admin_lib.php`), pas un repli.

## [OPTIMIZED]
- **Injection serveur `{{SITE:dotted.path|fallback}}`** ([dev_server.py](../dev_server.py) `_serve_html`/`_apply_site_placeholders`, [api/_html_server.php](../api/_html_server.php) `lumen_apply_site`) — le `<title>`, les `<meta description|keywords>` et la marque sont résolus **côté serveur** depuis `config/instance.json` (SEO-correct, sans flash), en amont de l'injection du nonce CSP. La valeur est HTML-échappée ; un `|fallback` inline garantit un rendu propre si la config manque. Twins Python/PHP en lockstep (comme `_csp_policy`/`lumen_csp_policy`). Résout le piège « double source » des titres : les `document.title = 'IRIBHM…'` en dur de [landing.js](../js/pages/landing.js) sont supprimés au profit de `InstanceConfig.applyHead()` (même source de vérité).
- **`_atomic_write` — mode POSIX optionnel** ([dev_server.py](../dev_server.py)) — nouveau paramètre `mode=` : `mkstemp` crée des temporaires 0600 (correct pour les secrets `api/`), mais un fichier `config/` PUBLIC doit être lisible par un serveur statique séparé (Apache servant un fichier écrit par php-fpm) → `mode=0o644` pour la config publique.
- **`_UPDATE_PROTECT` — config opérateur préservée** ([dev_server.py](../dev_server.py)) — `config/instance.json`, `config/theme.json`, `config/theme.css`, `config/legal.json`, `config/pages` ajoutés à la liste de protection de l'auto-updater : une mise à jour plateforme ne peut plus écraser la personnalisation de l'opérateur. `config/defaults/` (les gabarits neutres livrés) reste HORS protection pour rester mis à jour par les releases.
- **Substitution de tokens dans `I18n.t()`** ([js/core/i18n.js](../js/core/i18n.js)) — chaque chaîne de langue peut désormais référencer `{brand}`/`{specimen}`/… interpolés depuis `InstanceConfig.tokens()` (les params explicites du site d'appel priment). Remplacement par fonction (évite les pièges de motif `$`). Gardé : i18n fonctionne intact si `InstanceConfig` est absent. Permet de sortir les noms métier des `lang/*.json` sans les tripler en/fr/es.

## Notes
- **Vérification systématique — chemins Python ET PHP** : titre servi injecté depuis `config/instance.json` sur serveur Python (port de vérif) **et** hôte PHP (`php -S` + `router.php`) — placeholder `{{SITE:…}}` résolu côté serveur (aucun placeholder résiduel), nonce CSP toujours injecté, `data-instance` rempli côté client, `InstanceConfig.tokens()` peuplé, `/api/site.php?action=get` renvoie la config, `config/instance.json` servi en statique (200), **écriture sans auth rejetée en 401** sur les deux hôtes. Fonctions de persistance testées unitairement : roundtrip save/load, `publish` (draft→published), `reset` (→défaut), rejet de traversal (`../secret`, `pages/../../etc` → refusés). Aucune erreur console.
- **Portée de cette version** : socle uniquement. Une seule page (`index.html`) est câblée comme **implémentation de référence** (title/meta via placeholders serveur, `data-instance` pour marque/pied de page, chargement `InstanceConfig` avant i18n dans [landing.js](../js/pages/landing.js)). Le déploiement sur les 7 autres pages, la généralisation des clés i18n en/fr/es, et le découplage des hypothèses biologiques du JS (stats, facettes « stage », presets couleur de canaux) arrivent en v1.8.x (Objectif 1, phase de découplage).
- **Modèle de migration** : le `config/instance.json` committé = IRIBHM (l'instance vivante reste identique). Le builder de release exclura la config opérateur du zip et livrera le neutre — l'instance IRIBHM garde son identité via sa `config/` locale protégée (absente du manifest ⇒ jamais écrasée), à finaliser avec le marketplace/release.
- Conception & feuille de route complète : [DOCS/whitelabel/PLAN.md](../DOCS/whitelabel/PLAN.md).
