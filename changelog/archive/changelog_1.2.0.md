# Plateforme Web — v1.2.0

> **Internationalisation complète + plugins traduisibles.** Le système de langues passe en découverte dynamique (déposer `lang/<code>.json` l'active, comme pour les plugins) et chaque plugin embarque désormais son propre dossier `lang/`. La plateforme fusionne ces dictionnaires sous `plugins.<id>` dans l'arbre i18n, ce qui fait tomber « gratuitement » la règle de repli demandée : une langue présente côté plateforme mais absente d'un plugin retombe sur l'anglais **de ce plugin** uniquement ; une langue qu'un plugin embarque mais que la plateforme n'a pas n'est tout simplement jamais proposée. Le sélecteur de langue est généré à partir des langues réellement présentes. Toute la chaîne de textes utilisateur restante a été passée par i18n (viewer, tracking, compare, Download Center, Studio, panneau admin auparavant 100 % codé en français). Vérifié par tests Node vm + tests serveur + smoke test navigateur (sélecteur dynamique + bascule de langue live).

## [ADDED]
- **Dossier `lang/` par plugin** — chaque plugin (`js/modules/<placement>/<id>/lang/{en,fr,es}.json`) porte ses propres chaînes. La plateforme les charge via `I18n.loadPluginLang(id, path, langs)` ([js/core/i18n.js](../js/core/i18n.js)) au moment de `PluginRegistry.loadModules`, **avant** toute construction d'UI, et les greffe dans `_loaded[lang].plugins[id]`. Le code d'un plugin appelle `ctx.i18n.t('clé')` (façade scopée injectée par `PluginRegistry.initAll`, voir [js/core/plugin-registry.js](../js/core/plugin-registry.js)) → résolution `plugins.<id>.clé`.
- **Découverte dynamique des langues** — `I18n.discoverLanguages()` résout les locales de la **plateforme** selon le même repli hybride que les plugins : (1) endpoint live `GET /api/languages`, (2) manifeste statique `lang/manifest.json`, (3) défaut embarqué `['en','fr','es']` (plancher anti-crash, Rule 1.1). `I18n.getAvailableLanguages()` ne renvoie que ces locales — une langue propre à un plugin n'y figure jamais.
- **Endpoint serveur `/api/languages`** — `dev_server.py` scanne `lang/*.json`, garantit `en` en tête et **réécrit `lang/manifest.json`** à chaque appel (repli statique toujours à jour). Alias `/api/languages.php`. `_list_plugins` enrichit chaque `plugin.json` d'un `i18nLanguages` dérivé du scan de son dossier `lang/`.
- **Générateur autonome** — `tools/gen_lang_manifest.py` produit `lang/manifest.json` en standalone (déploiement statique sans `dev_server.py`), miroir de `tools/gen_plugins_manifest.py`.
- **Sélecteur de langue généré** — `Utils.populateLanguageMenu(onSwitch)` ([js/core/utils.js](../js/core/utils.js)) construit le menu `#lang-dropdown` depuis `I18n.getAvailableLanguages()` (drapeau + nom natif + état actif). Câblé sur `index.html` et `explorer.html` ; le hack codé en dur de `landing.js` (surcharge manuelle des `dropdown-item`) est supprimé. Déposer `lang/zh.json` ajoute « 🇨🇳 中文 » au menu sans toucher au HTML.
- **Registre de métadonnées de locales** `LANG_META` (natif, drapeau, `rtl`) pour `en/fr/es/zh/de/it/pt/nl/ja/ko/ru/ar` — une nouvelle langue connue s'affiche proprement, une inconnue retombe sur un libellé neutre. `dir=rtl` appliqué automatiquement pour les locales RTL.
- **`i18nLanguages` dans `plugin.json`** — liste explicite des locales embarquées par le plugin (évite les 404 ; tenue à jour par le scan serveur).

## [OPTIMIZED]
- **Repli par-plugin sans code dédié** — parce que les dictionnaires de plugins vivent dans le **même** arbre par-langue que la plateforme (sous `plugins.<id>`), tout le mécanisme existant (`I18n.t`, attributs `data-i18n*`, repli anglais de `t()`) fonctionne sur les clés de plugin sans cas particulier. Si un plugin n'embarque pas la locale active, `_loaded[lang].plugins[id]` est absent → `t()` retombe sur `_loaded.en.plugins[id]` : exactement la règle « langue manquante → anglais du plugin ».
- **Libellés toolbar/shaders résolus côté plugin** — `buildToolbarButtons` résout `i18nTitle`/`i18nAria` d'abord dans l'espace `plugins.<id>.*` (puis repli sur une clé plateforme legacy, puis `meta.name`) et estampille la bonne clé dans `data-i18n-title` pour la re-traduction au changement de langue. Le `<select>` de render-mode ([js/pages/viewer.js](../js/pages/viewer.js)) résout le nom du shader via `plugins.<id>.title`.
- **Re-rendu au changement de langue** — `PluginRegistry` s'abonne à `I18n.onLanguageChange` et invoque le hook optionnel `onLanguageChange()` des plugins (implémenté p.ex. par `measure-distance` pour re-peindre statut et liste).
