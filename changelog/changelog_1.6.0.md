# Plateforme Web — v1.6.0

> **Isolation des plugins tiers.** Un `index.js` de plugin ne peut plus s'exécuter avec les pleins privilèges de la page sans le consentement explicite de l'opérateur. Deux leviers : (1) un **gate de confiance default-deny** — le code non vérifié ne s'exécute jamais en page (seuls les plugins livrés dans la release signée, ou approuvés par l'opérateur et épinglés au hash de leur contenu, tournent in-page) ; (2) une **lane bac à sable** — un plugin approuvé « sandboxed » (ou tiers voulu) tourne dans une iframe null-origine sans accès au DOM, aux cookies, ni à l'API admin, communiquant uniquement via un pont RPC à capabilities étroites. Ce cycle bump la ligne mineure (Y) : nouveau sous-système de sécurité majeur.

## [ADDED]
- **Gate de confiance des plugins (default-deny)** ([js/core/plugin-trust.js](../js/core/plugin-trust.js), [dev_server.py](../dev_server.py), [js/core/plugin-registry.js](../js/core/plugin-registry.js)) — le serveur est l'autorité de confiance : il classe chaque plugin en `bundled` (fichiers présents dans le manifeste `version.json` de la release, match de **contenu** par sha256), `dev` (signal positif `.git`/`--dev-trust-local`, jamais l'absence de `version.json` — INV-3), `approved` (approbation opérateur épinglée au hash + jeu de capabilities), ou `untrusted` (défaut → non chargé, exclu de la découverte). Le client re-hash les **octets exacts** qu'il va exécuter et refuse tout mismatch (anti-TOCTOU, INV-2). Hash canonique (`lumen-plugin-trust/1`) partagé par les jumeaux JS/Python, validé byte-exact (CRLF/BOM) par [tests/plugin-trust-vector.json](../tests/plugin-trust-vector.json).
- **Exécution in-page par Blob-URL des octets hashés** ([js/core/plugin-registry.js](../js/core/plugin-registry.js) `_execTrustedInPage`) — un plugin trusté n'est plus injecté par `<script src>` (qui re-fetcherait des octets potentiellement différents) mais depuis un Blob des octets déjà hashés : les octets hashés == les octets exécutés.
- **Store d'approbation + endpoints** ([dev_server.py](../dev_server.py), [api/plugin-trust.json]) — `plugin_trust` (liste), `approve_plugin` (POST, auth **+ CSRF + ré-authentification mot de passe**, le serveur relit et re-hash lui-même, exige client==serveur — INV-4), `revoke_plugin`. Store jamais servi en HTTP, ajouté à `_UPDATE_PROTECT`, et **rejeté de tout artefact de release** (une release ne peut pré-approuver un plugin — INV-5).
- **Lane bac à sable** ([js/core/plugin-sandbox.js](../js/core/plugin-sandbox.js)) — un plugin `sandboxed` (tools action/toggle uniquement en v1) tourne dans une `<iframe sandbox="allow-scripts">` **sans** `allow-same-origin` (origine null : pas de DOM parent, pas de cookies, pas de fetch crédentialé) + CSP interne `default-src 'none'` / `connect-src 'none'` (pas d'exfiltration réseau, pas de Worker). Pont RPC : authentification par **identité de fenêtre** (`event.source`, jamais l'origine), jeton per-frame, discriminant de namespace `lumen-plugin` (jamais confondu avec `SYNC_*`), broker de capabilities (`toolbar.addButton`, `ui.toast`, `ui.download` gesture-gated, `viewer.getCanvasBlob`, `viewer.getInfo`, `viewer.setRenderMode`, `channels.getState`, `events.subscribe`) avec rate-limit token-bucket, heartbeat (kill au 3ᵉ pong manqué), compteur d'abus, et projections whitelist anti-pollution de prototype.
- **Plugin de référence sandboxé** ([js/modules/tools/screenshot-sandboxed/](../js/modules/tools/screenshot-sandboxed/)) — capture le canvas et le télécharge, entièrement dans l'iframe via capabilities (aucun accès à `ExportManager`/`document`/`ctx`), en contraste avec `tools/screenshot` in-page.
- **UX admin — approbation des plugins** ([js/pages/admin/tab-plugins.js](../js/pages/admin/tab-plugins.js)) — chaque plugin affiche son tier de confiance + hash ; un plugin `untrusted` propose Approuver (in-page / sandboxed, avec choix des capabilities) et Révoquer, avec ré-authentification.
- **CSP (Report-Only en v1.6)** — en-tête `Content-Security-Policy-Report-Only` posé par le serveur : surface les scripts/eval qui violeraient la politique stricte cible (`script-src 'self' 'nonce-…'`) **sans rien casser**, préparant le passage en enforcing (le vrai verrou d'exécution, INV-1) une fois les inline-scripts noncés.

## [OPTIMIZED]
- **Découverte annotée de confiance** ([dev_server.py](../dev_server.py) `_serve_plugins`) — `/api/plugins` exclut les plugins non fiables et joint à chacun un vouch `{tier,hash,mode,caps,files}` que le client re-vérifie, plus `devTrust`/`trustEpoch`.

## [FIXED] — issus de la revue adversariale pré-commit (14 défauts confirmés + corrigés)
- **Zombie iframe après timeout de spawn** ([js/core/plugin-sandbox.js](../js/core/plugin-sandbox.js)) — un plugin envoyant `ready` mais jamais `init-done` laissait une iframe vivante détenant ses capabilities après le timeout ; `spawn` appelle désormais `kill` (retrait `_hosts`/`_byId` + détachement), et le registry `kill` aussi en secours.
- **`.git` activait le dev-trust même exposé sur le réseau** ([dev_server.py](../dev_server.py), [api/_admin_lib.php](../api/_admin_lib.php)) — un checkout servi sur `0.0.0.0` faisait tourner in-page tout plugin déposé localement. Le dev-trust `.git` est désormais **gated au loopback** (Python : host localhost/127.0.0.1 ; PHP : `REMOTE_ADDR` loopback). Une prod (release, sans `.git`) reste fail-closed.
- **Chemin legacy fail-OPEN** ([js/core/plugin-registry.js](../js/core/plugin-registry.js)) — si `plugin-trust.js` ne chargeait pas, tous les plugins étaient injectés sans vérification. Désormais **fail-closed** (quarantaine) ; le chemin d'injection par URL non vérifiée est supprimé.
- **Origine non-sécurisée cassait le viewer** ([js/core/plugin-trust.js](../js/core/plugin-trust.js)) — sur `http://<IP-LAN>` (pas de `crypto.subtle`), tous les plugins devenaient `untrusted` → viewer sans shaders. Fallback : faire confiance au vouch serveur sans re-hash client (INV-2 relâché, averti), le serveur excluant toujours les non-fiables.
- **Approbation « sandbox » écrasée par le dev-trust** ([dev_server.py](../dev_server.py), PHP) — une approbation `sandboxed` (containment délibéré) prime désormais sur le dev-trust au lieu d'être exécutée in-page.
- **Flood `sys` contournait le rate-limit** ([js/core/plugin-sandbox.js](../js/core/plugin-sandbox.js)) — le budget token-bucket + compteur d'abus s'applique désormais à **tous** les messages entrants (pas seulement `req`), fermant un vecteur de DoS du thread de rendu.
- **Vérification `bundled` partielle sur host statique** ([js/core/plugin-trust.js](../js/core/plugin-trust.js)) — la branche sans-vouch énumère désormais **tous** les fichiers du manifeste (pas seulement index.js + plugin.json).
- **Store d'approbation servi en HTTP / commitable** ([api/.htaccess](../api/.htaccess), [.gitignore](../.gitignore)) — `.htaccess` deny désormais **tout** `api/*.json` ; `api/plugin-trust.json` + `quarantined-plugins.json` ajoutés au `.gitignore`.
- **Divergence de caps par défaut JS/Python/PHP** ([api/_admin_lib.php](../api/_admin_lib.php)) — le fallback caps d'un plugin sandboxé sans caps déclarées est aligné (`SANDBOX_DEFAULT_CAPS`, 3 caps).
- **Cap manquante + regex `</script>` + toggle** — plugin exemple : `ui.toast` déclaré ; garde `</script>` élargi aux variantes ; `button-state` routé en `sys`.

## Notes
- **Sécurité — franchise sur le périmètre v1.6** : le gate de confiance (le correctif racine) est complet et testé (parité de hash tri-implémentation, classification serveur). La lane bac à sable et l'UX d'approbation nécessitent une **vérification en navigateur réel** (messagerie iframe, CSP) que l'environnement de préview headless ne permet pas — à valider avant usage en production avec des plugins tiers réels. La CSP reste en **Report-Only** tant que les scripts inline des pages ne sont pas noncés ; l'enforcing est le verrou final (INV-1). L'authenticité par signature Ed25519 reste hors-scope (pas de crypto asymétrique en stdlib Python) — l'authenticité vient de HTTPS-vers-first-party + sha256, comme l'updater.
- **Tests** : parité du hash canonique (JS+Python, byte-exact CRLF/BOM), classification serveur (bundled/dev/approved/untrusted + épinglage), validation du broker RPC (identité de source, rate-limit, cap interdite, gesture-gate).
- Détail complet de la conception et des invariants : [DOCS/plugin-sandbox/SPEC.md](../DOCS/plugin-sandbox/SPEC.md).
